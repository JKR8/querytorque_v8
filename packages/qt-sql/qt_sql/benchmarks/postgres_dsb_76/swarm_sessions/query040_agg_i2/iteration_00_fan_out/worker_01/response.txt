## Part 1: Modified Logic Tree

```
QUERY: (single statement)
├── [~] CTE: date_filter
│   └── SCAN (date_dim) with filter on d_date range
├── [~] CTE: item_filter
│   └── SCAN (item) with filter on i_category and i_manager_id
├── [~] CTE: fact_join
│   ├── JOIN (catalog_sales ⋈ date_filter ON cs_sold_date_sk)
│   ├── JOIN (⋈ item_filter ON i_item_sk)
│   ├── JOIN (⋈ warehouse ON cs_warehouse_sk)
│   └── LEFT JOIN (catalog_returns ON cs_order_number AND cs_item_sk)
│   └── FILTER (cs_wholesale_cost range AND (cr_reason_sk=40 OR NULL))
├── [=] MAIN: aggregation
│   ├── AGG (GROUP BY w_state, i_item_id)
│   ├── SORT (w_state, i_item_id)
│   └── OUTPUT (w_state, i_item_id, sales_before, sales_after)
```

**Changes:**
- Added explicit CTEs for `date_filter` and `item_filter` to pre-filter dimension tables
- Rewrote `fact_join` from comma-separated joins to explicit `INNER JOIN`/`LEFT JOIN` syntax
- Preserved `LEFT JOIN` semantics with `(cr_reason_sk = 40 OR cr_reason_sk IS NULL)` filter
- Maintained all original literals and defensive `COALESCE` guards

**Changes:** Pre-filtered dimensions into CTEs and converted to explicit JOIN syntax. This enables PostgreSQL to build small hash tables from the dimension CTEs (30 rows from date_dim, few from item) and use hash joins with accurate cardinality estimates, while preserving LEFT JOIN semantics for catalog_returns.

**Expected speedup:** ~2.5-3x from better join planning and parallel execution enabled by avoiding CTE materialization fences.

**Runtime config rationale:**
- `max_parallel_workers_per_gather = 2`: Allows parallel scans of catalog_sales (large fact table)
- `work_mem = '16MB'`: Sufficient for hash tables from small dimension CTEs and one hash aggregation
- `hash_mem_multiplier = 2.0`: Gives hash joins extra headroom without overshooting memory budget
