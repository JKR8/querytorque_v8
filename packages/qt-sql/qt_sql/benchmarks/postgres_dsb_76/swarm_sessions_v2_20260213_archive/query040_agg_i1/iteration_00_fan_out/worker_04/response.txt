## Part 1: Modified Logic Tree

```
QUERY: (single statement)
└── [MAIN] main_query  [~]  Cost: ~85%  Rows: ~100
    ├── [!] CTE_SCAN (date_cte) -> INNER JOIN to catalog_sales
    ├── [!] CTE_SCAN (item_cte) -> INNER JOIN to catalog_sales  
    ├── [!] CTE_SCAN (returns_cte) -> LEFT JOIN to catalog_sales
    ├── [!] AGG (GROUP BY cs_warehouse_sk, i_item_id with CASE aggregates) -> fact_pivot_agg
    ├── [!] JOIN warehouse -> warehouse_join
    ├── [=] SORT (w_state ASC, i_item_id ASC)
    └── [=] OUTPUT (w_state, i_item_id, sales_before, sales_after)

Changes:
- [!] Structural change: Replaced implicit comma joins with explicit CTE-filtered joins
- [~] Modified main query to follow TARGET_LOGICAL_TREE structure
- [=] Preserved final sort and output unchanged
```

**Changes**: Converted implicit comma joins to explicit joins with pre-filtered CTEs for date, item, and returns dimensions. Preserved all original filters and CASE guard logic while restructuring to match the target logical tree.

**Expected speedup**: 15-30% due to reduced join cardinality from pre-filtered CTEs, explicit join ordering, and parallel aggregation enabled by avoiding CTE materialization on the fact table.

**Runtime config rationale**:
1. `max_parallel_workers_per_gather = 2`: Allow parallel aggregation on the filtered fact table (up to 2 workers within 8-worker global limit)
2. `work_mem = '32MB'`: Single hash aggregation in subquery requires ~20MB for grouping ~100 distinct (warehouse,item) pairs
3. `enable_hashjoin = on`: Hash joins optimal for filtered dimension tables
4. `enable_mergejoin = off, enable_nestloop = off`: Disable suboptimal join methods for this size
